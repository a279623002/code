# 面试高频问题笔记

> 整理项目面试中常见的高频问题，按 QPS、Agent、微服务、MySQL、Go 五个方向归类，附带面试标准答案。

## 一、QPS 相关问题

### 1. QPS 如何计算？

**一句话**：QPS = 总请求数 / 统计时间（秒）。

```
QPS = 总请求数 / 时间（秒）

例如：1 分钟处理 6 万次请求
QPS = 60000 / 60 = 1000
```

**常见统计方式**：

| 方式 | 工具/方法 |
|---|---|
| 网关日志 | Nginx / Kong / Gateway 日志分析 |
| 监控平台 | Prometheus + Grafana |
| 压测工具 | JMeter、wrk、ab、locust |
| 代码埋点 | 限流器/中间件计数 |

**延伸指标**：

| 指标 | 含义 |
|---|---|
| **TPS** | 每秒事务数，关注业务完成量 |
| **RT** | 响应时间 |
| **并发数** | 同时处理的请求数 |
| **成功率** | 成功请求占比 |

### 2. 哪个项目并发量大？QPS 多少？

**参考回答**：

我参与过的项目中，并发量较大的是 **算力调度系统**。该系统对接多个算力中心和训练任务，高峰期需要同时处理大量任务调度请求、资源查询请求和状态上报请求。

- 日常 QPS 大约在 **500 ~ 1000**
- 任务提交高峰期（如训练批次启动时）QPS 可达 **2000 ~ 3000**
- 瞬时峰值通过消息队列削峰，保证核心调度服务稳定

> 实际面试中可根据自己真实项目调整数字，重点是能讲清楚峰值场景和应对措施。
> 1. **你怎么知道 QPS 是这么多？**
>    - 生产环境通过 **Prometheus + Grafana** 监控接口调用量，按时间粒度统计
>    - 通过 **Nginx/网关访问日志** 离线分析
>    - 通过 **JMeter/wrk 压测** 得到系统容量上限
> 2. **峰值场景是什么？**
>    - 算力调度系统的峰值通常出现在**训练批次统一启动时**
>    - 大量任务同时提交、查询资源、上报状态
>    - 日常白天也有波峰波谷，夜间任务少
> 3. **有没有限流、降级、扩容预案？**
>    - 有：网关层做令牌桶限流，核心接口熔断保护
>    - 非核心查询接口可以降级返回缓存数据
>    - K8s HPA 配置 CPU/内存阈值自动扩容
> 4. **最慢接口/瓶颈在哪里？**
>    - 通常是任务调度接口，涉及 K8s API 调用和状态同步
>    - 优化方向：异步化、缓存资源状态、连接池优化
>
> 切忌只报一个数字，没有背景和应对思路。

### 3. 如果 QPS 突然暴涨怎么办？

**标准答案**：

QPS 暴涨时，核心思路是**先保命、再分析、再扩容、再优化**。

```
1. 限流降级
   ├── 网关层限流（令牌桶/漏桶）
   ├── 接口级熔断降级
   └── 非核心功能降级

2. 快速扩容
   ├── K8s HPA 自动扩实例
   ├── 数据库连接池扩容
   └── 缓存扩容

3. 削峰填谷
   ├── 请求入消息队列
   └── 异步处理

4. 定位原因
   ├── 看监控：QPS、RT、错误率、CPU、内存
   ├── 看日志：异常堆栈、慢查询
   └── 看流量来源：是否是爬虫、攻击、热点事件

5. 长期优化
   ├── 加缓存
   ├── SQL 优化
   ├── 异步化改造
   └── 分库分表
```

**面试加分项**：
- 提到 **预热** 和 **缓存击穿防护**
- 提到 **限流不是拒绝用户，而是保护系统**
- 提到 **压测确定系统容量基线**

### 4. QPS 和 TPS 有什么区别？

**标准答案**：

| 指标 | QPS | TPS |
|---|---|---|
| 全称 | Queries Per Second | Transactions Per Second |
| 关注点 | 每秒查询/请求数 | 每秒完成的事务数 |
| 粒度 | 一次 HTTP 请求 | 一组业务操作 |
| 关系 | 一个 TPS 可能包含多个 QPS | TPS 更偏向业务完成度 |

**举例**：

```
下单接口：用户点击一次"下单"
- 这是 1 个 QPS
- 但下单涉及：扣库存、生成订单、写日志、发消息
- 对业务来说这是 1 个 TPS

如果一次下单调用了 5 个内部接口，
那么 QPS = 5，TPS = 1
```

**面试一句话**：
> QPS 看请求量，TPS 看业务量；一个业务事务可能对应多个请求。

## 二、Agent 相关问题

### 1. 表结构写在 Prompt 里面吗？

**标准答案**：

**不会直接把完整表结构写进 Prompt**，原因：

1. 表结构太长，浪费 Token，容易超出上下文窗口
2. 表多了之后，模型容易选错表、选错字段
3. 直接暴露数据库结构有安全风险

**正确做法**：

| 方案 | 说明 |
|---|---|
| **Schema 摘要** | 只放业务相关表的字段名、类型、含义 |
| **按需加载** | 根据用户问题，只加载相关表的 Schema |
| **元数据管理** | 用专门的 Schema Store 管理，动态查询 |
| **示例驱动** | 给少量标准 SQL 示例，让模型举一反三 |
| **权限隔离** | 只暴露只读视图或脱敏后的结构 |

**实际落地**：

```
用户：查北京昨天的订单量

系统：识别到需要 "orders" 表
      加载 orders 表摘要：
      - city (string): 城市
      - order_time (datetime): 下单时间
      - amount (decimal): 金额

模型基于摘要生成 SQL
```

### 2. 用户历史对话怎么获取？

**标准答案**：

历史对话获取方式取决于**会话标识（session_id）** 和 **存储方案**。

```
用户请求
   │
   ▼
根据 user_id + session_id 查询对话记录
   │
   ▼
按时间排序，取最近 N 轮
   │
   ▼
放入 Prompt 上下文
```

| 存储方案 | 适用场景 |
|---|---|
| **数据库（MySQL/PostgreSQL）** | 需要长期保存、结构化查询 |
| **Redis** | 近期会话缓存，快速读取 |
| **向量库** | 长期记忆语义检索 |
| **本地内存** | 短期、单机会话 |

**优化**：
- 只取最近 N 轮（如 5~10 轮）
- 远期历史做摘要压缩
- 按话题分组，避免无关历史干扰

### 3. 怎么实现记忆？

**标准答案**：

Agent 记忆通常分三层：

```
┌─────────────────┐
│   短期记忆       │  当前对话上下文
│  Context Window │  放在 Prompt 里
├─────────────────┤
│   中期记忆       │  近期会话历史
│   Redis/DB      │  按 session 读取
├─────────────────┤
│   长期记忆       │  用户画像、偏好、历史摘要
│   向量库         │  语义检索召回
└─────────────────┘
```

| 记忆类型 | 存储 | 作用 |
|---|---|---|
| **短期记忆** | Prompt 上下文 | 保证当前多轮对话连贯 |
| **中期记忆** | Redis/DB | 同一个 session 的历史对话 |
| **长期记忆** | 向量库 | 跨 session 的用户偏好、事实 |

**实现方式**：
- 每次对话把用户输入和系统回复存入数据库
- 对历史对话做 Embedding，存入向量库
- 新对话时，先用用户问题检索相关历史记忆
- 把检索结果 + 近期对话一起放入上下文

**常见追问与回答**：

**Q：如果用户切换话题，短期记忆怎么处理？**

**答**：识别话题切换后，把旧话题的关键信息做摘要存入长期记忆或 session 上下文，新话题用新的短期记忆窗口。避免旧话题干扰当前回答。

**Q：短期记忆和长期记忆怎么配合？**

**答**：短期记忆保证当前对话连贯，长期记忆提供跨会话的用户画像和历史事实。每次请求时，先检索长期记忆，再拼接近期短期记忆，一起送给 LLM。

**Q：短期记忆存在哪里？**

**答**：存在本次请求的 Prompt 里，服务端不单独持久化。如果需要跨请求保留，靠 session_id 从 Redis/DB 读取近期对话后拼入 Prompt。

### 4. 如果 SQL 执行报错怎么办？有没有自动修正机制？

**标准答案**：

SQL 执行报错时，要走**分级处理**机制：

```
SQL 执行报错
   │
   ▼
解析错误类型
   │
   ├── 语法错误 → 把错误信息回传给 LLM，要求修正 SQL
   │
   ├── 表/字段不存在 → 重新加载 Schema，再次生成
   │
   ├── 权限错误 → 返回标准化提示，不走自动修正
   │
   ├── 超时/连接错误 → 重试，失败则降级
   │
   └── 数据问题 → 返回"查询无结果"，禁止编造
```

**自动修正机制**：

| 机制 | 说明 |
|---|---|
| **错误回传** | 把数据库报错信息返回给 LLM，让模型重新生成 |
| **重试次数限制** | 最多重试 2~3 次，防止死循环 |
| **Schema 校验** | 执行前先校验表和字段是否存在 |
| **参数校验** | 防止 SQL 注入、非法参数 |
| **只读限制** | 默认只允许 SELECT，禁止 DML/DDL |
| **兜底话术** | 修正失败时返回"查询失败，请联系管理员" |

**面试重点**：
> 自动修正不是万能的，必须加**重试上限**和**兜底机制**，防止模型死循环。

### 5. RAG 支持纯文本还是图片等多模态？怎么实现？

**标准答案**：

RAG 可以支持**纯文本、图片、表格、音频、视频**等多种模态。

```
文本：直接分片 → Embedding → 向量检索
图片：OCR/视觉模型提取描述 → 文本 Embedding
表格：结构化解析 → 文本描述或专用表格向量模型
音频：ASR 转文本 → 文本 Embedding
视频：抽帧/ASR → 文本或视频向量
```

| 模态 | 处理方式 |
|---|---|
| **纯文本** | Markdown/纯文本分片 + 文本 Embedding |
| **图片** | OCR 提取文字 / 多模态模型生成描述 |
| **表格** | 转成 Markdown 表格或 JSON，再做向量化 |
| **PDF** | 提取文字 + OCR 识别图片中的文字 |
| **音频/视频** | ASR 转写文本后再处理 |

**多模态检索实现**：
- 文本走文本向量模型（如 BGE）
- 图片走视觉语言模型（如 CLIP、Qwen-VL）
- 建立统一的检索索引，查询时按模态路由

### 6. Chunk 分块策略

**标准答案**：

Chunk 是把长文档切成小段的过程，分块策略直接影响检索质量。

| 策略 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| **固定长度** | 简单、快速 | 可能切断语义 | 通用文档 |
| **按段落/句子** | 语义完整 | 段落长短不一 | 文章、报告 |
| **重叠分片** | 避免信息断裂 | 存储增加 | 大多数场景 |
| **按标题/章节** | 保留文档结构 | 需要预处理 | 手册、政策文件 |
| **语义分块** | 语义边界清晰 | 成本高 | 高质量问答 |

**最佳实践**：

```
通用方案：固定长度 + 重叠窗口 + 按标题边界优化

参数参考：
- 切片大小：300~800 字符
- 重叠比例：10%~20%
- 按标题优先切分，避免标题和正文被切开
```

**面试加分项**：
- 提到 **父子分块（parent-child chunking）**：小块用于精准召回，大块用于上下文组装
- 提到 **表格、代码等特殊内容单独处理**

### 7. 怎么评估 RAG 召回率？怎么提升？

**标准答案**：

#### 评估指标

| 指标 | 含义 | 公式 |
|---|---|---|
| **召回率（Recall）** | 相关文档被找回的比例 | 召回相关数 / 总相关数 |
| **精确率（Precision）** | 召回结果中相关的比例 | 相关数 / 召回总数 |
| **F1** | 精确率和召回率的调和平均 | 2 * P * R / (P + R) |
| **TopK 命中率** | 正确答案是否在前 K 个结果中 | - |
| **NDCG** | 考虑排序质量的综合指标 | - |

#### 评估方法

```
1. 准备测试集：标准问题 + 期望答案 + 相关片段
2. 运行检索，得到 TopK 结果
3. 计算 Recall、Precision、F1、命中率
4. 分析 BadCase，持续优化
```

#### 提升召回率的方法

| 方向 | 具体手段 |
|---|---|
| **检索策略** | 向量 + 关键词混合检索 |
| **Query 增强** | 同义词扩展、Query 改写、意图补全 |
| **分片优化** | 调整 chunk 大小、增加重叠、按语义切分 |
| **Embedding 优化** | 用领域微调后的 Embedding 模型 |
| **扩大 TopK** | 粗召回多取一些，交给 Rerank 精排 |
| **重排优化** | 用 Rerank 模型二次排序 |
| **知识库补全** | 补充缺失内容、清理过期噪声 |

## 三、微服务相关问题

### 1. 微服务拆分策略，为什么这样拆？

**标准答案**：

微服务拆分没有银弹，常见策略有四种：

| 拆分维度 | 说明 | 例子 |
|---|---|---|
| **按业务领域（DDD）** | 按业务能力拆分，最推荐 | 用户服务、订单服务、库存服务 |
| **按数据归属** | 按数据实体拆分 | 用户数据、商品数据、日志数据 |
| **按功能职责** | 按技术职责拆分 | 网关服务、文件服务、消息服务 |
| **按团队边界** | 康威定律，团队独立维护 | 小组 A 负责订单，小组 B 负责支付 |

**拆分原则**：

```
1. 高内聚：同一服务内部功能紧密相关
2. 低耦合：服务间依赖尽量少
3. 独立部署：每个服务可独立发版
4. 独立扩展：按负载单独扩容
5. 数据独立：每个服务有自己的数据库
```

**项目中的拆分示例（算力调度系统）**：

```
算力调度系统：
├── 用户服务（认证、权限）
├── 资源服务（算力节点、集群管理）
├── 任务服务（任务提交、状态跟踪）
├── 调度服务（任务编排、K8s 调用）
├── 监控服务（指标采集、告警）
└── 网关服务（统一入口、限流鉴权）
```

**为什么这样拆**：
- 用户服务变化少，独立部署不影响核心调度
- 调度服务压力大，可单独扩容
- 监控服务日志量大，独立避免影响业务
- 每个服务有独立数据库，避免单点耦合

### 2. 本地缓存占用过大怎么办？

**标准答案**：

本地缓存过大时，从**容量控制、过期策略、分布式缓存、监控告警**四个方向处理。

| 方案 | 说明 |
|---|---|
| **设置容量上限** | 如 Caffeine maximumSize = 10000 |
| **设置过期时间** | 过期自动清理，避免无限增长 |
| **LRU/LFU 淘汰** | 热点保留，冷门自动淘汰 |
| **分级缓存** | 本地缓存 + Redis 分布式缓存 |
| **内存监控** | 超过阈值告警，及时排查 |
| **缓存预热控制** | 避免启动时大量加载 |

**缓存结构建议**：

```
用户请求
   │
   ▼
本地缓存（Caffeine）命中？→ 返回
   │
   ▼
Redis 命中？→ 返回并写入本地缓存
   │
   ▼
数据库查询 → 写入 Redis + 本地缓存
```

**面试加分项**：
- 提到 **本地缓存适合读多写少、一致性要求不高的数据**
- 提到 **缓存失效时的雪崩、击穿、穿透防护**

### 3. 一共有多少个实例节点？

**参考回答**：

我参与的算力调度系统采用 Kubernetes 部署，微服务总实例数根据环境不同：

| 环境 | 实例数 |
|---|---|
| 开发环境 | 每个服务 1 个 Pod |
| 测试环境 | 核心服务 2 个 Pod |
| 生产环境 | 核心服务 3~5 个 Pod，非核心服务 2 个 Pod |

**核心服务**：网关、用户服务、调度服务、任务服务
**非核心服务**：监控服务、文件服务

> 面试时根据自己的真实项目回答，重点是能讲清楚**为什么这样分配**。
> 1. **为什么核心服务 3~5 个 Pod？**
>    - 根据压测结果：单 Pod 压测 QPS 约 500，生产日常 1000~1500，留 2~3 倍冗余
>    - 调度服务是核心链路，3 个 Pod 保证高可用，避免单点
>    - 任务服务高峰期扩容到 5 个，应对批量任务提交
> 2. **有没有 HPA 自动扩缩容策略？**
>    - 有，配置 CPU 使用率 > 70% 持续 1 分钟自动扩容
>    - 最大副本数限制，防止无限扩容打崩依赖
>    - 缩容有冷却时间，避免抖动
> 3. **非核心服务为什么可以少？**
>    - 监控服务主要是日志写入和指标查询，流量低
>    - 文件服务有对象存储兜底，2 个 Pod 足够
>    - 失败不影响核心业务流程，可降级
> 4. **实例数变化过没有？**
>    - 有，训练批次启动时调度服务从 3 扩到 5
>    - 大促/发布前会提前手动扩容并压测验证
> 5. **有没有做资源隔离？**
>    - 给每个 Pod 配置 CPU/Memory Request 和 Limit
>    - 核心服务单独节点池部署，避免被其他服务抢占
>    - 关键链路设置超时和熔断
>

### 4. 了解过 Consul/etcd 服务注册吗？

**标准答案**：

了解过，二者都是**服务注册与发现**组件。

#### Consul

| 特性 | 说明 |
|---|---|
| 功能 | 服务注册、发现、健康检查、KV 存储 |
| 协议 | HTTP/gossip |
| 优点 | 功能丰富、界面友好 |
| 缺点 | 部署复杂、性能不如 etcd |

#### etcd

| 特性 | 说明 |
|---|---|
| 功能 | 分布式键值存储、强一致性、Watch 机制 |
| 协议 | Raft |
| 优点 | 高性能、强一致、K8s 原生支持 |
| 缺点 | 只提供 KV，需自行做服务发现封装 |

#### 对比

| 维度 | Consul | etcd |
|---|---|---|
| 一致性 | 最终一致 | 强一致 |
| 性能 | 中 | 高 |
| 功能 | 服务发现 + 配置 + 健康检查 | 存储 + Watch |
| K8s 生态 | 可用 | 原生支持 |

**项目中使用**：
- 如果项目用 K8s，通常直接用 K8s Service + etcd（K8s 自身用 etcd）
- 如果项目不用 K8s，可以用 Consul 或 Nacos 做服务发现

## 四、MySQL 相关问题

### 1. 索引建立设计

**标准答案**：

索引设计要遵循 **"先查后建、覆盖高频、避免过多"** 的原则。

#### 建立索引的原则

| 原则 | 说明 |
|---|---|
| **WHERE 字段建索引** | 高频查询条件优先 |
| **JOIN 字段建索引** | 关联字段必须有索引 |
| **ORDER BY / GROUP BY 字段建索引** | 避免文件排序 |
| **区分度高的字段建索引** | 性别、状态等低区分度字段不适合单独建索引 |
| **避免冗余索引** | 已有 (a,b) 联合索引，不需要单独给 a 建索引 |
| **控制索引数量** | 索引过多影响写入性能 |

#### 联合索引最左前缀

```sql
-- 联合索引 (a, b, c)
WHERE a = 1              -- 走索引
WHERE a = 1 AND b = 2    -- 走索引
WHERE a = 1 AND c = 3    -- 只走 a
WHERE b = 2              -- 不走索引
```

#### 项目中如何设计

```sql
-- 订单表
CREATE TABLE orders (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id BIGINT NOT NULL,
    order_no VARCHAR(32) NOT NULL,
    status TINYINT NOT NULL,
    create_time DATETIME NOT NULL,
    INDEX idx_user_id (user_id),
    INDEX idx_order_no (order_no),
    INDEX idx_status_create_time (status, create_time)
);
```

**设计思路**：
- `user_id`：按用户查订单，高频
- `order_no`：订单号精确查询，唯一
- `(status, create_time)`：后台按状态和时间范围查询

### 2. 如何优化 SQL？

**标准答案**：

SQL 优化从**索引、查询写法、表结构、架构**四个层面入手。

#### 索引层面

- 加合适索引
- 删除冗余索引
- 用覆盖索引减少回表

#### 查询写法层面

```sql
-- 避免 SELECT *
SELECT id, name FROM user WHERE id = 1;

-- 避免在索引列上用函数
-- 错误
SELECT * FROM user WHERE YEAR(create_time) = 2024;
-- 正确
SELECT * FROM user WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';

-- 避免隐式类型转换
-- 错误
SELECT * FROM user WHERE phone = 13800138000;  -- phone 是字符串
-- 正确
SELECT * FROM user WHERE phone = '13800138000';
```

#### 表结构层面

- 选择合适的数据类型
- 避免大字段频繁查询
- 垂直拆分/水平拆分

#### 架构层面

- 加缓存（Redis）
- 读写分离
- 分库分表
- 异步处理

#### 分析工具

```sql
-- 查看执行计划
EXPLAIN SELECT * FROM user WHERE id = 1;

-- 查看慢查询
SHOW VARIABLES LIKE 'slow_query_log%';
```

**面试一句话**：
> 优化 SQL 先看执行计划，确认是否走索引；再看查询写法，避免函数、隐式转换、全表扫描；最后考虑缓存、读写分离、分库分表。

## 五、Go 相关问题

### 1. 为什么 GMP 要这样设计？

**标准答案**：

GMP 是 Go 的并发调度模型：

| 组件 | 含义 | 作用 |
|---|---|---|
| **G（Goroutine）** | 协程 | 用户态轻量线程 |
| **M（Machine）** | 操作系统线程 | 真正执行代码 |
| **P（Processor）** | 逻辑处理器 | 管理 G 的队列和上下文 |

```
G 任务 ──► P 本地队列 ──► M 执行
              │
              ▼
         全局队列（所有 P 共享）
```

#### 为什么要这样设计？

| 原因 | 说明 |
|---|---|
| **Goroutine 极轻量** | 初始栈只有 2KB，比线程小得多 |
| **避免线程频繁切换** | 用户态调度，减少内核态开销 |
| **多核并行** | P 数量默认等于 CPU 核数，充分利用多核 |
| **本地队列减少锁竞争** | 每个 P 有自己的 G 队列，避免全局锁 |
| **负载均衡** | 工作窃取机制，空闲 M 从其他 P 偷任务 |

#### 相比传统线程池的优势

```
传统线程：1 个线程 ≈ 1MB 栈空间，切换成本高
Goroutine：2KB 起，可轻松创建百万级
```

#### 为什么要加 P 这一层？

```
如果没有 P，G 直接和 M 绑定：
- G 创建销毁频繁，M 也要频繁创建销毁
- 多核扩展复杂
- 调度器锁竞争激烈

加入 P 后：
- P 持有 G 队列，M 从 P 取任务
- M 可以复用，不用频繁创建
- 本地队列 + 全局队列 + 工作窃取，调度高效
```

**面试一句话**：
> GMP 用两层调度把用户态轻量协程和内核线程解耦，G 轻量、P 管理队列、M 负责执行，兼顾高并发、低开销和多核利用。

## 六、一句话总结

- **QPS**：QPS 看请求量，TPS 看业务量；暴涨时先限流降级、再扩容削峰
- **Agent**：表结构不直接写 Prompt，历史对话按 session 查询，记忆分短/中/长三层
- **RAG**：支持多模态，分片用固定+重叠+标题优化，评估看 Recall/Precision/F1
- **微服务**：按业务领域拆分，本地缓存过大用容量上限+过期+LRU+分布式缓存
- **MySQL**：索引建在高频 WHERE/JOIN/ORDER BY 字段，优化先看执行计划
- **Go**：GMP 用用户态协程降低线程开销，P 解耦调度与执行，提升多核效率
