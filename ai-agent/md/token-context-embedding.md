# Token / 上下文 / Embedding 基础

> 一句话：Token 是模型的计算单位，上下文窗口是短期记忆容量，Embedding 是长期语义存储。

---

## 一、Token 基础

### 1. 什么是 Token？

**一句话**：Token 是大模型的最小计算单位，模型不认识文字，只认识 Token。

### 2. Token 与字符的关系

| 语言 | 大约比例 |
|---|---|
| 中文 | 1 个汉字 ≈ 2 Token |
| 英文 | 1 个单词 ≈ 1~2 Token |
| 数字/标点 | 单独算 Token |

### 3. Token 为什么重要？

| 影响 | 说明 |
|---|---|
| 上下文容量 | Token 越多，能放的内容越少 |
| 推理成本 | 按 Token 计费 |
| 推理速度 | Token 越多，响应越慢 |

---

## 二、上下文窗口（Context Window）

### 1. 定义

**一句话**：上下文窗口是模型单次能处理的最大 Token 总量，包括输入 + 输出。

### 2. 上下文超限的问题

- 系统 Prompt 规则丢失
- 历史对话被截断
- 关键工具结果缺失
- 输出逻辑混乱

### 3. 优化方案

| 方案 | 说明 |
|---|---|
| Prompt 精简 | 删除冗余话术 |
| 滑动窗口 | 只保留近期对话 |
| 摘要压缩 | 远期历史做摘要 |
| 关键信息置顶 | 核心规则放前面 |
| 预留输出余量 | 别让输入占满窗口 |

---

## 三、Embedding（向量嵌入）

### 1. 定义

**一句话**：Embedding 是把文本语义转换成高维数字向量的技术。

### 2. 核心原理

```
语义相近的文本 → 向量距离近
语义无关的文本 → 向量距离远
```

### 3. 在 Agent 中的作用

| 作用 | 说明 |
|---|---|
| 知识库检索 | 从海量文档中召回相关内容 |
| 长期记忆 | 存储历史对话、用户偏好 |
| 意图识别 | 辅助判断用户问题类型 |

---

## 四、三者协作关系

```
用户提问
   │
   ▼
Embedding 检索相关知识
   │
   ▼
筛选精简内容
   │
   ▼
放入上下文窗口
   │
   ▼
LLM 推理生成答案
```

| 角色 | 作用 |
|---|---|
| **Context Window** | 短期记忆，容量有限 |
| **Embedding** | 长期语义存储，突破窗口限制 |
| **LLM** | 最终推理和生成 |

---

## 五、面试高频问题

### Q1：Token 和字符有什么区别？

**答**：Token 是模型的计算和计费单位，字符是文本展示单位。中文 1 个汉字大约 2 Token。

### Q2：上下文窗口不足怎么办？

**答**：精简 Prompt、滑动窗口、摘要压缩、关键信息置顶、预留输出余量。

### Q3：Embedding 为什么能检索相似内容？

**答**：语义相近的文本向量化后距离近，通过相似度计算可以召回相关内容。

### Q4：Token、上下文、Embedding 怎么协作？

**答**：Embedding 从海量知识中召回相关内容，筛选后放入有限的上下文窗口，LLM 再基于这些内容推理回答。

### Q5：为什么长文档不能直接塞进上下文？

**答**：上下文窗口有限，直接塞入会导致截断、Token 成本高、信息密度低，关键信息被淹没。

## 六、项目口述模板

> 你项目中怎么处理 Token 和上下文问题？

我在项目中通过 Embedding + 上下文管理来解决。长文档先做语义分片，向量化存储到向量库；用户提问时通过 Embedding 检索召回相关片段，再用 Rerank 精排过滤噪声；最后只把高相关片段放入上下文窗口，既突破了窗口限制，又控制了 Token 成本。同时用滑动窗口和摘要压缩管理多轮对话历史，避免上下文爆炸。

---

## 七、一句话总结

- **Token：模型的计算和计费单位**
- **上下文窗口：短期记忆，容量有限**
- **Embedding：长期语义存储，突破窗口限制**
- **三者协作：Embedding 找知识 → 上下文装精华 → LLM 做推理**
