## 八、Token / Embedding / Context Window 核心面试专项

**适用方向**：Agent上下文优化、长文本处理、向量检索、模型性能与成本优化

**使用说明**：口语化精简标准答案，贴合Agent工程落地，可直接背诵面试

### 1、什么是 Token？Token 和 汉字、字符的区别是什么？

**标准答案**：

Token是大模型的**最小计算单元**，是模型训练和推理的基础分片单位，不是简单的字符或汉字。中文大概**1个汉字≈2个Token**，英文是单词、词根、标点、空格都会被拆成独立Token。

核心区别：字符是纯文本展示单位，Token是模型**理解、计算、计费、上下文限制**的唯一标准。模型不认识文字，只认识Token序列，所有窗口限制、计费、推理速度都基于Token统计。

### 2、Token 为什么会影响 Agent 效果和成本？

**标准答案**：

主要影响三点：

第一是**上下文容量**，Token超限会导致截断、丢历史记忆、丢Prompt规则，直接让Agent跑偏、失控；

第二是**推理成本**，输入输出Token直接决定计费，越长成本越高；

第三是**推理速度**，Token越多，模型解码步数越多，Agent响应延迟越高，体验越差。

### 3、什么是 Context Window（上下文窗口）？核心作用是什么？

**标准答案**：

上下文窗口是大模型**单次对话能容纳的最大Token总量**，包含系统Prompt、历史对话、用户当前提问、工具返回结果、模型输出内容全部内容。

对Agent来说，它的核心作用是**维持任务连续性**。Agent多轮工具调用、迭代思考、多轮复盘都依赖上下文留存，窗口不足会直接导致规则丢失、任务断裂、遗忘前置指令，出现越权、答非所问、重复调用工具等问题。

### 4、上下文窗口超限，Agent 会出现哪些典型问题？

**标准答案**：

常见五类工程问题：

1\. 遗忘系统Prompt约束，防幻觉、防越权规则失效，可控性崩塌；

2\. 丢失历史对话信息，多轮任务断层，需求理解不完整；

3\. 文本强制截断，关键参数、工具返回结果缺失，导致答案错误；

4\. 重复提问、重复调用工具，出现无效循环调用；

5\. 输出逻辑混乱、格式错乱，稳定性大幅下降。

### 5、工程中如何解决 Context Window 不足、长会话超限问题？

**标准答案**：

我在Agent项目中采用四层优化方案，由轻到重依次落地：

第一，Prompt轻量化，精简冗余话术，模块化拆分提示词，只保留核心业务规则；

第二，动态会话管理，采用滑动窗口保留近期对话，对远期历史做**摘要压缩、关键信息萃取**；

第三，记忆分层设计，区分短期会话记忆、长期业务记忆，不重要的对话直接丢弃；

第四，长文本拆分\+增量问答，超长文档分片处理，避免单次灌入超限内容。

### 6、什么是 Embedding（向量嵌入）？核心原理是什么？

**标准答案**：

Embedding是将**自然语言文本转化为高维稠密向量**的技术。简单说，就是把文字的语义，转换成一组数字数组，让机器可以量化理解文本的语义相似度。

核心原理：语义相近的文本，向量空间距离更近；语义无关的文本，向量距离更远。它不关注字面匹配，只关注**语义匹配**，是RAG检索、知识库问答、Agent长文本记忆的核心基础。

### 7、Embedding 在 Agent 体系中的核心作用是什么？

**标准答案**：

主要解决大模型**上下文有限、知识陈旧、幻觉严重**的三大痛点，核心用途有三点：

第一，长文本知识库检索，通过Embedding相似度匹配，召回用户问题对应的真实文档，替代一次性灌入全文；

第二，长期记忆管理，将历史对话、业务知识向量化存储，突破上下文窗口限制，实现永久记忆；

第三，问题聚类与意图识别，辅助Agent精准判断用户需求，提升问答相关性。

### 8、请说下 Embedding、LLM、Context Window 三者的协作关系？（高频综合题）

**标准答案**：

三者是互补协作、互相兜底的关系：

Context Window 是模型的**临时短时容量**，负责承载当前轮次的Prompt、对话和工具结果，容量有限；

Embedding 是**永久语义存储与检索**，突破窗口限制，把海量外部知识、历史记忆做向量化召回；

LLM 负责**最终推理、思考、工具调用、答案生成**。

整体流程就是：用户提问→Embedding检索相关知识→筛选精简内容塞入上下文窗口→LLM基于真实资料推理输出，完美解决Agent长文本、大知识库场景的准确性和上下文不足问题。

### 9、Embedding 为什么会出现检索不准？常见原因和优化方案？

**标准答案**：

常见原因：字面不一致但语义一致、文本分片不合理、向量模型不匹配、噪声数据过多、问句泛化性强。

优化手段分四层：

1\. 优化切片策略，固定合理分片长度、重叠分片，避免切断语义；

2\. 问题改写，对用户口语化问题做标准句式重构，提升检索匹配度；

3\. 多路召回，结合向量检索\+关键词检索，互补兜底；

4\. 重排序Rerank，对召回结果做二次精排，过滤无效内容，提升输入上下文质量。

### 10、Agent 中「Token成本、上下文上限、Embedding检索精度」如何做工程平衡？

**标准答案**：

核心原则是：**用Embedding扩容量、用上下文保精度、用Token控制成本**。

首先，海量知识不进上下文，全部通过Embedding向量检索筛选，突破窗口限制；

其次，控制单次召回Token数量，只放入高相关内容，避免超限和冗余；

最后，通过摘要、分片、重排序，在**精度不下降的前提下最小化Token消耗**，实现Agent性能、效果、成本的最优平衡。

### 11、区分：模型上下文窗口、输入Token、输出Token的工程差异？

**标准答案**：

上下文窗口是**输入\+输出的总Token上限**；输入Token是传给模型的Prompt、历史、检索内容；输出Token是模型思考和回答的内容。

工程落地中最容易踩坑的是：只计算输入Token，忽略模型输出占用的窗口，导致多轮Agent迭代过程中突然截断、规则丢失，我在项目中会**预留固定输出Token余量**，规避超限风险。

### 12、长文本Agent，你是如何结合 Embedding \+ 上下文优化提升准确率的？（项目面试题）

**口述模版（直接背）**：

我在做文档问答Agent时，遇到的核心问题是长文档无法全部塞入上下文，直接截断会丢失关键信息，导致答案不准、幻觉高发。

我的解决方案是，先对全文做语义分片，通过Embedding将所有片段向量化存储。用户提问时，先通过向量相似度检索召回高相关片段，再通过Rerank重排序过滤低质量内容，精简有效信息。最后将筛选后的少量高质量内容灌入模型上下文，既突破了Context Window的容量限制，又控制了Token成本，同时保证模型每一轮推理都基于真实文档内容，大幅提升了Agent输出的准确性和可控性。
