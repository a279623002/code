## 二、参数 Schema 设计（工程核心重点）

### 3、什么是 Function Schema？作用是什么？

**标准答案**：

Schema是对工具函数的**结构化描述模板**，是LLM理解工具、生成正确参数的唯一依据。标准Schema包含函数名称、功能描述、入参参数列表、参数类型、是否必填、参数释义、枚举约束等内容。

核心作用：约束模型调用行为，避免参数乱填、漏填、错填，让模型严格按照业务规范生成结构化调用参数，保证工具调用的准确性和可控性。

### 4、一套标准、可落地的 Function Schema 包含哪些字段？

**标准答案**：

我工程中使用的完整Schema包含6个核心模块：

1\. **function name**：函数名称，唯一标识，简洁语义化；

2\. **description**：详细功能描述，精准定义适用场景、不适用场景；

3\. **parameters**：入参结构体，包含所有请求参数；

4\. **param type/required**：参数数据类型、是否必填；

5\. **param description/enum**：参数释义、取值范围、枚举值、格式约束；

6\. **default/example**：默认值、标准入参示例，降低模型理解成本。

### 5、Schema 设计的核心原则？如何避免参数调用错误？

**标准答案**：

遵循四大工程原则：

**第一，语义极简精准**：描述不模糊，明确工具能做什么、不能做什么，防止模型误用；

**第二，参数约束穷尽**：必填、类型、长度、枚举、格式全部写死，不给模型自由发挥空间；

**第三，场景边界明确**：标注禁止调用场景，避免多余调用、错误调用；

**第四，示例辅助兜底**：复杂参数配套标准样例，大幅提升参数准确率。

### 6、多工具场景下，如何设计 Schema 避免模型选错工具？

**标准答案**：

核心是**场景差异化隔离**：

第一，每个工具的description强绑定专属业务场景，杜绝语义重叠；

第二，差异化参数体系，不同工具参数结构、必填字段做区分；

第三，在全局Prompt中补充工具选择优先级规则，明确不同需求对应的优先调用工具；

第四，通过Schema注释标注互斥工具，禁止同时调用冲突函数。

## 三、Function Call 完整调用链路（面试高频）

### 7、详细讲一遍 Agent Function Call 的完整执行链路？

**标准答案**：

完整分为6个闭环步骤，我全程落地的工程链路：

**第一步：需求接收与决策**：Agent接收用户提问，结合系统Prompt和工具Schema，自主判断是否需要调用工具，还是直接回答；

**第二步：结构化参数生成**：模型根据Schema约束，解析用户意图，提取、补全、校验入参，输出标准FunctionCall结构化数据；

**第三步：本地参数校验**：业务层做二次校验，检查参数缺失、格式错误、越权参数，拦截非法调用；

**第四步：工具/API 实际执行**：本地程序解析函数名和参数，发起API请求、数据库查询或脚本执行；

**第五步：结果回传与上下文拼接**：将工具返回结果、执行状态封装为标准消息，写入对话上下文；

**第六步：二次推理生成答案**：LLM基于工具真实返回结果，结合用户需求，生成最终自然语言回答，完成闭环。

### 8、什么是多轮Function Call？什么场景需要循环调用？

**标准答案**：

多轮Function Call就是Agent**一次任务连续多次调用工具**，自主迭代、分步完成复杂任务。

适用场景：需要分步查询、参数依赖、多工具组合、条件分支执行的任务。比如先查用户信息、再查订单数据、最后汇总统计，单轮无法完成，需要多轮循环调用。

Agent会根据上一轮工具返回结果，自主判断是否需要继续调用、切换工具、补充参数，直到任务完成再输出最终答案。

## 四、错误处理与工程容错（加分核心）

### 10、Function Call 常见的报错类型有哪些？

**标准答案**：

工程中高频五类异常：

1\. **工具选择错误**：选错函数、无关调用、重复调用；

2\. **参数异常**：参数缺失、类型错误、格式不规范、参数越界；

3\. **接口执行异常**：超时、网络失败、接口报错、权限不足；

4\. **空结果异常**：工具执行成功但无数据返回；

5\. **循环死调用**：多轮调用无法终止，无限重复请求同一工具。

### 11、针对参数错误、漏参错参，你的分层处理方案？

**标准答案**：

我采用三层容错机制：

**第一层：前置Prompt约束**：通过Schema强约束、参数示例、必填提示，从源头减少参数错误；

**第二层：本地代码校验**：调用工具前做参数正则、类型、非空、范围校验，提前拦截错误，不允许脏参数进接口；

**第三层：智能补参与重试**：检测到缺参时，Agent自动询问用户补齐必要信息，或根据上下文推断补全，合法后再重试调用。

### 12、接口超时、报错、网络异常，如何做兜底处理？

**标准答案**：

分为重试机制\+用户兜底\+日志溯源三步：

第一，**有限重试**：针对瞬时网络波动，配置2\~3次低频次重试，避免单次抖动导致任务失败；

第二，**异常兜底话术**：重试失败后，禁止模型编造数据，统一输出合规异常提示，清晰告知用户工具调用失败、原因及后续建议；

第三，**异常日志埋点**：记录函数名、入参、报错信息、时间戳，用于后续BadCase迭代优化。

### 13、如何解决 Function Call 死循环、无限调用问题？

**标准答案**：

通过三重限制彻底杜绝死循环：

第一，**轮次上限控制**：单任务限制最大工具调用次数，超过阈值强制终止任务；

第二，**重复调用拦截**：上下文校验，相同函数\+相同参数重复调用直接拦截，判定为无效循环；

第三，**终止条件Prompt约束**：明确工具返回有效结果后、或多次无结果时，必须停止调用，直接整理结果回复用户。

### 14、工具返回空数据时，怎么防止Agent幻觉编造？

**标准答案**：

我在系统Prompt中做强制约束：

工具返回空、无匹配数据、查询结果为空时，**禁止推测、禁止脑补、禁止模糊概括**。必须如实告知用户当前无对应数据，不做任何虚假补充，同时停止继续无效调用，直接结束本轮工具流程。
