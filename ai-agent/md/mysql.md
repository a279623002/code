# MySQL 面试笔记

> MySQL 是关系型数据库主力，面试核心围绕引擎、索引、事务、锁、主从复制、分布式事务。调度系统、机器人管控平台等业务数据都依赖 MySQL。

---

## 一、InnoDB vs MyISAM

| 特性 | InnoDB | MyISAM |
|---|---|---|
| 事务 | 支持 | 不支持 |
| 行锁 | 支持 | 不支持（只支持表锁） |
| 外键 | 支持 | 不支持 |
| 崩溃恢复 | 支持（redo log） | 不支持 |
| 索引类型 | 聚簇索引 | 非聚簇索引 |
| 适用场景 | 高并发 OLTP | 读多写少、日志分析 |

**结论**：MySQL 5.5 之后默认 InnoDB，生产环境基本都用 InnoDB。

## 二、索引类型

### 1. 按数据结构

| 类型 | 特点 |
|---|---|
| **B+Tree 索引** | 默认，适合范围查询和等值查询 |
| **Hash 索引** | 只支持等值查询，Memory 引擎支持 |
| **Full-Text 索引** | 全文检索 |
| **R-Tree 索引** | 空间数据索引 |

### 2. 按逻辑

| 类型 | 特点 |
|---|---|
| **主键索引** | 唯一、非空，一张表一个 |
| **唯一索引** | 值唯一，允许 null |
| **普通索引** | 无唯一约束 |
| **联合索引** | 多个字段组合 |
| **覆盖索引** | 索引包含查询所需全部字段 |

### 3. 联合索引最左前缀

```sql
CREATE INDEX idx_a_b_c ON t(a, b, c);

-- 能用索引
WHERE a = 1
WHERE a = 1 AND b = 2
WHERE a = 1 AND b = 2 AND c = 3
WHERE a = 1 AND c = 3      -- a 能用，c 不能

-- 不能用索引
WHERE b = 2
WHERE c = 3
WHERE b = 2 AND c = 3
```

## 三、索引失效场景

1. **违反最左前缀**
2. **对索引字段做函数/运算**
   ```sql
   WHERE YEAR(create_time) = 2024  -- 失效
   WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01'  -- 有效
   ```
3. **like 以 `%` 开头**
   ```sql
   WHERE name LIKE '%张%'  -- 失效
   WHERE name LIKE '张%'   -- 有效（范围扫描）
   ```
4. **隐式类型转换**
   ```sql
   WHERE phone = 13800138000  -- phone 是 varchar，会转换失效
   ```
5. **索引字段参与计算**
   ```sql
   WHERE id + 1 = 100  -- 失效
   WHERE id = 99       -- 有效
   ```
6. **OR 条件中部分字段无索引**
7. **!=、<>、NOT IN** 可能失效（取决于数据分布和优化器）

## 四、索引优化

### 1. 优化原则

- 为 WHERE、JOIN、ORDER BY、GROUP BY 字段建索引
- 选择性高的字段放联合索引左边
- 避免冗余索引
- 用小字段做主键，减少二级索引大小
- 利用覆盖索引减少回表

### 2. Explain 分析

```sql
EXPLAIN SELECT * FROM t WHERE a = 1;
```

| 字段 | 含义 |
|---|---|
| `type` | 访问类型：system > const > eq_ref > ref > range > index > ALL |
| `key` | 实际使用的索引 |
| `rows` | 扫描行数，越小越好 |
| `Extra` | Using index（覆盖索引）、Using where、Using filesort（需要排序优化） |

## 五、聚簇索引 vs 非聚簇索引

### 1. 聚簇索引（InnoDB）

```
主键索引（聚簇索引）：
┌─────────────┬──────────────────────────┐
│ 主键 id      │ 行数据（data）            │
├─────────────┼──────────────────────────┤
│     1       │ name=A, age=20, ...      │
│     2       │ name=B, age=25, ...      │
└─────────────┴──────────────────────────┘

特点：叶子节点存整行数据
```

### 2. 非聚簇索引（二级索引）

```
普通索引：
┌─────────────┬──────────────────────────┐
│ 索引字段 name│ 主键 id                   │
├─────────────┼──────────────────────────┤
│     A       │     1                    │
│     B       │     2                    │
└─────────────┴──────────────────────────┘

特点：叶子节点存主键，需要回表查整行数据
```

### 3. 回表 vs 覆盖索引

- **回表**：先查二级索引得到主键，再查聚簇索引
- **覆盖索引**：查询字段全在索引里，不用回表

```sql
-- 覆盖索引示例
CREATE INDEX idx_name_age ON user(name, age);
SELECT name, age FROM user WHERE name = 'A';  -- 不用回表
```

## 六、为什么索引用 B+树而不是 B树？

| 对比 | B 树 | B+树 |
|---|---|---|
| 数据存储 | 每个节点都存数据 | 只有叶子节点存数据 |
| 叶子节点 | 无链表 | 叶子节点双向链表 |
| 查找效率 | 不稳定 | 稳定（都到叶子） |
| 范围查询 | 需要中序遍历 | 直接叶子链表遍历 |
| 树高 | 更高 | 更低（一个节点存更多索引） |

**B+树优势**：
1. 树高低，IO 次数少
2. 叶子节点链表，范围查询快
3. 非叶子节点只存索引键，一页能存更多索引
4. 查询性能稳定

## 七、事务 ACID

| 特性 | 含义 | InnoDB 如何实现 |
|---|---|---|
| **A 原子性** | 要么全成功，要么全失败 | undo log |
| **C 一致性** | 数据从一种一致状态到另一种一致状态 | 其他三个特性保证 |
| **I 隔离性** | 事务之间互不干扰 | MVCC + 锁 |
| **D 持久性** | 提交后数据不丢失 | redo log + binlog |

## 八、事务隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 |
|---|---|---|---|
| READ UNCOMMITTED | 可能 | 可能 | 可能 |
| READ COMMITTED | 不可能 | 可能 | 可能 |
| REPEATABLE READ（默认） | 不可能 | 不可能 | 不可能（InnoDB 解决） |
| SERIALIZABLE | 不可能 | 不可能 | 不可能 |

### 三种问题

| 问题 | 说明 | 解决 |
|---|---|---|
| **脏读** | 读到其他事务未提交的数据 | READ COMMITTED 及以上 |
| **不可重复读** | 同一事务内两次读取结果不同 | REPEATABLE READ |
| **幻读** | 同一事务内两次查询，行数不同 | REPEATABLE READ + 间隙锁 |

### InnoDB 如何解决幻读？

- **快照读**：普通 SELECT 走 MVCC，读取事务开始时的快照，不会幻读
- **当前读**：`SELECT ... FOR UPDATE` 加间隙锁，阻止其他事务插入

## 九、锁

### 1. 按粒度

| 锁 | 说明 |
|---|---|
| **行锁** | 锁某一行，InnoDB 支持 |
| **表锁** | 锁整张表，MyISAM 支持 |
| **页锁** | 锁一页，BDB 引擎 |

### 2. 行锁类型

| 锁 | 说明 |
|---|---|
| **记录锁（Record Lock）** | 锁某条记录 |
| **间隙锁（Gap Lock）** | 锁索引之间的间隙，防止幻读 |
| **临键锁（Next-Key Lock）** | 记录锁 + 间隙锁 |
| **意向锁** | 表级锁，表示事务将要加行锁 |

### 3. 按功能

| 锁 | 说明 |
|---|---|
| **共享锁 S** | `SELECT ... LOCK IN SHARE MODE`，可读不可写 |
| **排他锁 X** | `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`，不可读写 |

### 4. 死锁

**原因**：两个事务互相等待对方持有的锁。

**解决**：
- 统一加锁顺序
- 尽量减小事务范围
- 使用乐观锁（version 字段）
- 设置超时：`innodb_lock_wait_timeout`

## 十、主从复制

### 1. 架构

```
        ┌─────────┐
        │  Master │ ← 写请求
        │  主库    │
        └────┬────┘
             │ binlog
    ┌────────┼────────┐
    ↓        ↓        ↓
┌───────┐ ┌───────┐ ┌───────┐
│Slave 1│ │Slave 2│ │Slave 3│ ← 读请求
└───────┘ └───────┘ └───────┘
```

### 2. 复制方式

| 方式 | 说明 |
|---|---|
| **异步复制** | Master 不等待 Slave 确认，速度快，可能丢数据 |
| **半同步复制** | Master 等至少一个 Slave 确认，平衡可靠和性能 |
| **全同步复制** | 等所有 Slave 确认，慢，少用在 MySQL |

### 3. binlog 格式

| 格式 | 说明 |
|---|---|
| STATEMENT | 记录 SQL 语句 |
| ROW | 记录每行变化，推荐 |
| MIXED | 混合模式 |

## 十一、分布式事务

### 1. 2PC（两阶段提交）

```
阶段 1：准备
    协调者 ──→ 询问所有参与者能否提交
    参与者 ──→ 执行本地事务，锁定资源，返回 Yes/No

阶段 2：提交/回滚
    全部 Yes → 协调者发 Commit
    任意 No  → 协调者发 Rollback
```

**问题**：协调者单点故障、同步阻塞、脑裂。

### 2. 3PC（三阶段提交）

增加 CanCommit 预询阶段，减少阻塞，但实现复杂。

### 3. 柔性事务（TCC / Saga）

| 方案 | 说明 |
|---|---|
| **TCC** | Try-Confirm-Cancel，业务层实现 |
| **Saga** | 长事务拆分，失败时补偿 |
| **本地消息表** | 异步保证最终一致性 |
| **Seata AT** | 自动补偿，阿里开源 |

## 十二、分布式锁

### 1. 基于 MySQL 实现

```sql
CREATE TABLE distributed_lock (
    lock_name VARCHAR(64) PRIMARY KEY,
    owner VARCHAR(64),
    expire_time TIMESTAMP
);
```

```sql
-- 加锁
INSERT INTO distributed_lock (lock_name, owner, expire_time)
VALUES ('order_lock', 'node1', DATE_ADD(NOW(), INTERVAL 30 SECOND));

-- 释放锁
DELETE FROM distributed_lock WHERE lock_name = 'order_lock' AND owner = 'node1';
```

**缺点**：依赖 DB 可用性，无自动续期。

### 2. 基于 Redis 实现（更常用）

```redis
set lock:order:1001 request_id nx ex 30
```

### 3. 基于 ZooKeeper/etcd 实现

利用临时顺序节点，释放锁时删除节点，监听者竞争。

## 十三、面试高频问题

### Q1：InnoDB 和 MyISAM 的区别？

**答**：InnoDB 支持事务、行锁、外键、聚簇索引、崩溃恢复；MyISAM 不支持事务和行锁，索引非聚簇，适合读多写少。

### Q2：索引为什么用 B+树？

**答**：B+树只有叶子节点存数据，非叶子节点更"瘦"，树高更低；叶子节点链表，范围查询效率高；查询性能稳定。

### Q3：聚簇索引和非聚簇索引的区别？

**答**：聚簇索引叶子节点存整行数据；非聚簇索引叶子节点存主键，需要回表。InnoDB 主键索引是聚簇索引，二级索引是非聚簇索引。

### Q4：什么是回表？什么是覆盖索引？

**答**：回表是查二级索引得到主键后再查聚簇索引；覆盖索引是查询字段全在索引中，不需要回表。

### Q5：MVCC 是什么？

**答**：多版本并发控制，通过 undo log 和 read view 实现。快照读不加锁，读取历史版本，实现高并发下的读写不阻塞。

### Q6：怎么避免死锁？

**答**：统一加锁顺序、减小事务粒度、用乐观锁、设置锁等待超时。

### Q7：主从延迟怎么解决？

**答**：读写分离后强制走主库、半同步复制、增加从库性能、业务容忍短暂不一致。

---

## 十四、一句话总结

- **引擎**：生产用 InnoDB，支持事务和行锁
- **索引**：B+树，联合索引注意最左前缀，避免函数/隐式转换导致失效
- **事务**：ACID，默认 REPEATABLE READ，MVCC + 锁保证隔离性
- **锁**：行锁、间隙锁、临键锁，死锁用统一顺序 + 小事务规避
- **主从**：异步/半同步复制，binlog ROW 格式
- **分布式事务**：2PC/3PC/TCC/Saga，根据场景选
- **分布式锁**：MySQL 可实现，但 Redis/etcd 更常用
