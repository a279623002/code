# Agent 三大架构模式

> 一句话：Tool-use 只会执行，ReAct 边想边做，Planning 先规划后执行。

---

## 一、三种模式总览

| 模式 | 核心思想 | 复杂度 | 适用场景 |
|---|---|---|---|
| **Tool-use** | 单次意图识别 + 单次工具调用 | 低 | 查天气、查订单等单点任务 |
| **ReAct** | 思考 → 行动 → 观察 → 迭代 | 中 | 多轮工具调用、动态适配 |
| **Planning** | 先全局规划，再分步执行 | 高 | 复杂长流程、强依赖任务 |

```
Tool-use：   输入 → 选工具 → 执行 → 输出（一次结束）
ReAct：      输入 → 思考 → 行动 → 观察 → 思考 → 行动 → 输出（循环）
Planning：   输入 → 全局拆解 → 子任务1 → 子任务2 → ... → 汇总输出
```

---

## 二、Tool-use 模式

### 1. 定义

**一句话**：Tool-use 是最简单的 Agent 形态，只做一次工具调用，直接返回答案。

### 2. 流程

```
用户：帮我查北京天气
 │
 ▼
LLM 判断需要调用 weather 工具
 │
 ▼
生成参数：{"city": "北京"}
 │
 ▼
调用天气接口
 │
 ▼
返回结果给 LLM
 │
 ▼
LLM 组织语言：北京今天晴，25℃
```

### 3. 优缺点

| 优点 | 缺点 |
|---|---|
| 架构简单 | 不能做多步骤任务 |
| 响应快 | 无法自我纠错 |
| Token 消耗低 | 遇到失败不会重试 |
| 稳定性高 | 无法处理模糊需求 |

---

## 三、ReAct 模式

### 1. 定义

**一句话**：ReAct = Reason（思考）+ Act（行动），边思考边行动，根据结果迭代。

### 2. 核心四步

```
Thought（思考）：是否需要工具？调哪个？参数是什么？
    │
    ▼
Action（行动）：调用工具
    │
    ▼
Observation（观察）：接收工具返回
    │
    ▼
Iteration（迭代）：任务是否完成？继续还是修正？
```

### 3. 示例

```
用户：帮我查张三的最新订单金额

思考：需要查用户信息，再查订单
行动：调用 get_user(name="张三")
观察：{user_id: 1001}

思考：已拿到 user_id，查订单
行动：调用 get_orders(user_id=1001)
观察：[{order_id: 2001, amount: 199}]

思考：已拿到订单金额，可以回答
输出：张三最新订单金额是 199 元
```

### 4. 优缺点

| 优点 | 缺点 |
|---|---|
| 能做多步骤任务 | 长任务容易跑偏 |
| 能自我纠错 | 没有全局规划 |
| 动态适配 | 可能无限循环 |
| 工业界最常用 | 步骤多易混乱 |

---

## 四、Planning 模式

### 1. 定义

**一句话**：Planning 先制定完整任务蓝图，再按规划逐步执行。

### 2. 两层架构

```
┌─────────────────┐
│   Plan 规划层    │  拆解任务、排优先级、识别依赖
│  （全局大脑）     │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ Execute 执行层   │  用 ReAct 逐个完成子任务
│  （动作单元）     │
└─────────────────┘
```

### 3. 流程示例

```
用户：帮我写一篇行业分析报告

规划层：
1. 搜索行业最新数据
2. 整理主要竞品信息
3. 分析市场趋势
4. 生成报告大纲
5. 撰写完整报告

执行层：
- 子任务1：调用搜索工具
- 子任务2：调用竞品分析工具
- 子任务3：调用数据分析工具
- 子任务4：调用大纲生成
- 子任务5：调用报告生成
```

### 4. 优缺点

| 优点 | 缺点 |
|---|---|
| 全局可控 | 成本高、耗时长 |
| 适合长任务 | 简单任务冗余 |
| 步骤不易遗漏 | 对模型能力要求高 |
| 可动态修正计划 | 实现复杂 |

---

## 五、三种模式对比

| 维度 | Tool-use | ReAct | Planning |
|---|---|---|---|
| 思考能力 | 无 | 局部思考 | 全局思考 |
| 执行能力 | 单次 | 多次迭代 | 按规划执行 |
| 纠错能力 | 无 | 有 | 有 |
| 响应速度 | 快 | 中 | 慢 |
| Token 成本 | 低 | 中 | 高 |
| 适用任务 | 单点查询 | 中等动态任务 | 复杂长流程 |

## 六、面试高频问题

### Q1：Tool-use 和 ReAct 的区别？

**答**：Tool-use 是单次调用，无迭代无纠错；ReAct 是思考 + 行动 + 观察的循环，能做多步骤任务并自我纠错。

### Q2：ReAct 长任务为什么会跑偏？

**答**：ReAct 只有局部视野，没有全局规划。步骤一多容易遗忘目标、重复调用、遗漏子任务。

### Q3：Planning 模式为什么适合复杂任务？

**答**：Planning 先把任务拆解成蓝图，明确步骤、依赖、终止条件，再分步执行，避免 ReAct 的局部混乱。

### Q4：实际项目怎么选型？

**答**：简单任务用 Tool-use，中等动态任务用 ReAct，复杂长流程用 Planning + ReAct。

### Q5：Planning + ReAct 怎么配合？

**答**：Planning 负责顶层任务拆解和进度管控，ReAct 负责每个子任务的落地执行，遇到异常再回传规划层调整。

---

## 七、项目口述模板

> 你项目中的 Agent 是什么架构模式？

我项目中采用分层架构选型：简单单点查询用 Tool-use，比如查天气、查状态；多轮工具组合用 ReAct，比如查用户信息再查订单；复杂长流程任务用 Planning + ReAct，比如生成行业报告、自动化办公流程。这样既能保证简单任务快速响应，又能让复杂任务可控、可追溯。

---

## 八、一句话总结

- **Tool-use：只会做，不会想**
- **ReAct：边想边做，最常用**
- **Planning：先想后做，做大事**
- **选型原则：按任务复杂度分层，不要一刀切**
