# Agent面试｜三大架构模式 Tool\-use / ReAct / Planning 原理、边界与工程取舍（独立完整版）

**文档定位**：纯独立专项文档，只聚焦工业级三大Agent架构模式，解决面试高频核心提问：原理、机制、适用场景、局限性、横向取舍、架构选型

**核心结论前置**：Tool\-use 只会调用、不会思考迭代；ReAct 边思考边行动、适合中等动态任务；Planning 先规划后执行、适合复杂长流程任务

## 一、整体架构总览（面试开篇必答）

### 1、Agent 三大主流架构模式分别是什么？核心分层定位？

**标准答案**：

目前工业落地的 Agent 架构分为由浅到深、由轻到重的三种范式：**Tool\-use 工具调用模式、ReAct 推理交互模式、Planning 全局规划模式**。

三者定位完全分层：

Tool\-use：基础能力层，解决「单点工具调用」问题，无自主推理迭代能力；

ReAct：动态推理层，解决「多步骤动态自适应执行」问题，具备实时思考\+行动闭环；

Planning：全局统筹层，解决「长周期、高复杂、强依赖任务」的全局调度问题。

## 二、Tool\-use 工具调用模式（基础版Agent）

### 2、Tool\-use 实现原理与核心机制？

**标准答案**：

Tool\-use 是最基础的 Agent 形态，核心机制是**单次意图识别 \+ 单次工具调用 \+ 直接输出结果**，无多轮思考、无迭代、无自我纠错。

完整执行链路：用户输入需求 → LLM 识别所需工具 → 抽取、校验入参 → 生成 FunctionCall → 调用外部工具/接口/知识库 → 获取结果 → LLM 整理话术直接返回。

核心特点：**无循环、无反思、无动态决策，一次调用即结束**。

### 3、Tool\-use 适用边界与优势？

**标准答案**：

**适用场景**：任务单一、步骤唯一、无需多轮联动的简单查询类任务。例如：查天气、查订单、单次知识库检索、简单数据统计。

**核心优势**：架构极简、耗时短、Token 消耗低、稳定性极高、几乎不会死循环、运维成本低。

### 4、Tool\-use 核心局限性（致命短板）？

**标准答案**：

第一，**不支持多步骤任务**，无法自主连续调用多个工具完成复合任务；

第二，**无纠错能力**，工具调用失败、参数错误、结果不满足需求时，不会重试、不会修正；

第三，**无动态适配能力**，无法根据返回结果调整下一步策略，完全依赖用户原始输入；

第四，**无法处理模糊、残缺需求**，缺参、需求不明确时无法自主追问和补全执行。

## 三、ReAct 推理交互模式（工业标准Agent）

### 5、ReAct 实现原理与核心机制？

**标准答案**：

ReAct（Reason \+ Act）是目前企业落地最广的标准 Agent 架构，核心机制是**思考、行动、观察、迭代的闭环循环**，实现真正的知行合一。

核心四步闭环机制：

1\. Thought 思考：模型基于当前上下文分析信息是否充足、是否需要调用工具、调用哪个工具、参数是否正确；

2\. Act 行动：执行工具调用、外部请求、知识检索；

3\. Observation 观察：接收外部真实返回结果；

4\. Iteration 迭代：模型基于最新结果重新决策，判断任务是否完成、是否需要继续调用、是否需要修正参数，直至任务闭环。

核心特点：**边想边做、动态决策、实时纠错、自适应迭代**。

### 6、ReAct 适用边界与优势？

**标准答案**：

**适用场景**：中等复杂度、多步骤、流程不固定、需要动态适配的任务。例如：多轮知识问答、多工具组合查询、问题排查、简单数据分析、动态需求处理。

**核心优势**：具备自我纠错能力，能处理模糊需求、残缺参数、执行异常；无需预设固定流程，适配性强；落地成本适中，效果远优于单纯 Tool\-use。

### 7、ReAct 核心局限性？

**标准答案**：

第一，**无全局规划视野**，只有局部最优，容易陷入局部循环、无效重试；

第二，**长任务极易跑偏**，步骤一多容易遗忘目标、遗漏子任务、逻辑混乱；

第三，**执行不可控**，迭代次数不可控，容易出现无限轮工具调用，消耗资源；

第四，不适合强流程、强依赖、高精密的复杂业务任务。

## 四、Planning 全局规划模式（高阶复杂Agent）

### 8、Planning 规划模式实现原理与核心机制？

**标准答案**：

Planning 架构核心机制是**规划与执行解耦**，遵循「先全局蓝图规划，后分步落地执行」的核心逻辑，彻底解决 ReAct 长任务失控问题。

分为双层核心机制：

**第一层：规划层（Plan）**

接收复杂需求后，先拆解完整子任务清单、梳理执行先后顺序、识别任务依赖、定义终止条件、预判风险，生成结构化任务执行蓝图，不做任何实际执行操作。

**第二层：执行层（Execute）**

严格按照规划蓝图，逐个子任务通过 ReAct 模式落地执行，每完成一步同步进度，遇到异常、数据缺失、需求变更时，回传给规划层动态更新蓝图，再继续执行。

核心特点：**全局统筹、结构化执行、进度可控、动态修正、杜绝无效迭代**。

### 9、Planning 适用边界与优势？

**标准答案**：

**适用场景**：长周期、多步骤嵌套、强依赖、高精密、流程复杂的重量级任务。例如：自动化办公流水线、复杂数据分析报告、多Agent协同任务、项目拆解落地、复杂业务流程审批。

**核心优势**：全局视野、任务不遗漏、执行有序、可管控、可追溯；解决 ReAct 长任务跑偏、循环、失控问题；适合企业级复杂业务落地。

### 10、Planning 核心局限性？

**标准答案**：

第一，**成本高、耗时久**，需要额外一次完整规划推理，Token 开销最大；

第二，**简单任务冗余严重**，简单查询用规划架构属于过度设计；

第三，对模型能力要求高，低能力模型容易出现规划拆解错误，导致整体执行失效。

## 五、三大模式横向对比与工程取舍（面试核心加分项）

### 11、Tool\-use / ReAct / Planning 完整取舍逻辑？

**标准答案**：

**场景取舍原则：按任务复杂度分层选型，绝不一套架构通用所有场景**

**① 简单单点任务 → 选用 Tool\-use**

单次查询、无需迭代、流程固定，追求速度与稳定性；避免 ReAct 多余迭代、避免 Planning 过度设计。

**② 中等动态任务 → 选用 ReAct**

多步骤、流程不固定、需要动态纠错适配，追求灵活度与自适应能力；平衡成本与智能性，是绝大多数业务场景的最优解。

**③ 复杂长周期任务 → 选用 Planning\+ReAct 组合**

多依赖、多嵌套、长流程、强管控需求；上层规划控全局，下层 ReAct 做落地，兼顾有序性和灵活性。

### 12、三大模式核心能力一句话总结（面试抢答）

**标准答案**：

**Tool\-use**：只会执行不会思考，轻量化单点工具调用，无迭代无纠错；

**ReAct**：边思考边行动，动态迭代自适应，适配绝大多数中等复杂业务；

**Planning**：先规划后执行，全局统筹可控，解决复杂长流程任务失控问题。
