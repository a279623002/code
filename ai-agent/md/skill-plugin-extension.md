## 一、核心总览（面试第一必问：三者层级关系）

### 1、请简述 Agent 中 Skill、Plugin、Extension 的层级与核心定位？

**标准答案**：

三者是Agent**由内到外、从核心到拓展**的分层能力扩展机制，层级从高到低、粒度从粗到细：

**Skill（技能）**：是Agent**业务能力单元**，面向用户场景，是最上层的业务能力封装，比如查询订单、数据分析、文档总结；

**Plugin（插件）**：是Agent**工具执行单元**，面向功能调用，承载Skill的具体执行逻辑，可被多个Skill复用；

**Extension（扩展）**：是Agent**框架底层钩子**，面向框架生命周期，用于扩展框架原生能力、修改底层流程，不直接面向业务功能。

简单总结：**Skill 做业务、Plugin 做功能、Extension 改框架**。

## 二、Skill 机制 专项面试题

### 2、什么是 Agent Skill（技能）？核心作用是什么？

**标准答案**：

Skill是Agent**面向终端业务的最小完整能力单元**，是对用户可见的业务技能。它整合了意图识别、参数解析、工具调用、结果整理、话术输出的完整链路，是用户直观感知到的Agent能力。

核心作用是**业务能力封装与场景化落地**，把零散的工具接口，封装成可直接解决用户问题的完整业务功能，让Agent具备场景化服务能力。

### 3、Skill 的内部组成结构是什么？

**标准答案**：

一套标准Skill包含五个核心模块：

第一，技能元信息：名称、功能描述、适用场景、禁用条件；

第二，意图匹配规则：用户触发该技能的问句特征、关键词、语义规则；

第三，参数抽取模板：需要提取的用户入参、缺参追问逻辑；

第四，依赖Plugin列表：绑定需要调用的底层插件/工具；

第五，结果后置处理：数据格式化、话术统一、异常兜底、输出组装。

### 4、Skill 机制的核心优势？解决什么工程问题？

**标准答案**：

第一，**业务解耦**：不同场景能力独立Skill管理，互不干扰，迭代更灵活；

第二，**能力复用**：底层通用Plugin可被多个Skill复用，避免重复开发；

第三，**快速拓展**：新增业务场景只需新增Skill，无需改动Agent核心框架；

第四，**可管控可灰度**：支持单个Skill开启、关闭、权限控制、灰度上线、版本迭代。

## 三、Plugin 机制 专项面试题

### 5、什么是 Agent Plugin（插件）？和 Skill 的区别？

**标准答案**：

Plugin是Agent**底层通用功能执行单元**，是纯粹的工具能力，只负责完成单一、具体的原子操作，比如HTTP请求、数据库查询、文件解析、代码执行、向量检索。

它和Skill的核心区别：

Plugin是**原子工具、无业务语义**，只做执行，不做业务逻辑组装；

Skill是**业务场景、有完整语义**，通过组合多个Plugin，实现完整用户需求。

从属关系：**Skill 依赖 Plugin 执行，Plugin 为 Skill 提供底层能力支撑**。

### 6、标准 Plugin 的工程结构与运行机制？

**标准答案**：

标准插件包含四层结构：

一是插件描述Schema：定义插件名称、功能、入参、出参、参数约束；

二是核心执行逻辑：原子功能代码，比如接口调用、数据处理；

三是前置校验逻辑：参数校验、权限校验、风控拦截；

四是后置兜底逻辑：异常捕获、重试机制、空结果处理、日志埋点。

运行机制是：Agent调度中心根据Skill需求，匹配对应Plugin，自动传参执行，返回标准化结果，供上层Skill组装输出。

### 7、Plugin 插件化架构的核心价值？

**标准答案**：

第一，**功能原子化**：统一拆解通用能力，避免代码冗余；

第二，**热插拔拓展**：支持动态新增、删除、启停插件，无需重启Agent服务；

第三，**统一管控**：所有工具调用统一走插件层，统一鉴权、审计、限流、容错；

第四，**解耦框架**：核心框架与业务工具完全解耦，提升系统稳定性和可维护性。

## 四、Extension 机制 专项面试题

### 8、什么是 Agent Extension（底层扩展）？核心定位？

**标准答案**：

Extension是**Agent框架底层的生命周期钩子扩展机制**，属于最内层的框架能力，不对外提供业务功能，也不被Skill直接调用。它的核心作用是拦截、修改、增强Agent原生框架的运行流程。

简单说：Plugin是扩展「Agent能做什么」，Extension是扩展「Agent怎么运行」。

### 9、Extension 常见落地场景有哪些？

**标准答案**：

工程中常用的Extension场景分为六类：

第一，请求前置扩展：用户输入清洗、敏感词过滤、意图预处理；

第二，Prompt扩展：动态拼接Prompt、规则注入、模板替换；

第三，调度扩展：工具调用优先级修改、路由规则拦截；

第四，结果后置扩展：输出内容脱敏、格式统一、话术修正；

第五，日志监控扩展：全链路埋点、BadCase采集、指标统计；

第六，安全风控扩展：权限拦截、防注入、防逃逸校验。

### 10、Extension 和 Plugin 的本质区别？（高频坑题）

**标准答案**：

第一，层级不同：Plugin是**业务工具层**，面向功能执行；Extension是**框架底层层**，面向流程增强；

第二，触发方式不同：Plugin由业务Skill主动调用，按需执行；Extension跟随框架生命周期自动触发，全局生效；

第三，作用不同：Plugin新增Agent业务能力；Extension优化、拦截、改造Agent原有运行流程；

第四，复用范围不同：Plugin单业务可选择性使用；Extension全局所有会话、所有Skill统一生效。

## 五、三者综合对比与架构面试题（压轴必问）

### 11、一句话区分 Skill、Plugin、Extension？

**标准答案**：

**Skill**：用户可见的业务技能，解决「用户要什么」；

**Plugin**：底层原子工具能力，解决「具体怎么做」；

**Extension**：框架生命周期扩展，解决「流程怎么优化管控」。

### 12、请完整描述 Agent 三层能力扩展架构的执行链路？

**标准答案**：

完整执行链路从外到内、从上到下：

首先，**Extension底层拦截**：用户请求进入框架后，先走前置Extension，完成清洗、风控、预处理；

其次，**Skill业务匹配**：Agent识别用户意图，命中对应业务Skill；

最后，**Plugin原子执行**：Skill调度依赖的各类Plugin，完成工具调用、数据处理，最终经过后置Extension统一处理后输出结果。

整体形成：**Extension管控流程 → Skill承接业务 → Plugin落地执行**的闭环架构。

### 13、什么场景用 Plugin？什么场景用 Extension？如何选型？

**标准答案**：

如果是**新增具体业务功能、原子工具能力**，比如新增查数据、解析文件、联网搜索，选Plugin；

如果是**修改全局流程、统一管控、底层增强**，比如全局脱敏、统一Prompt、全局风控、日志埋点，选Extension；

如果是**封装完整用户场景、组合多个工具能力**，选Skill。
