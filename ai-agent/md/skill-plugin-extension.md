# Agent 三层扩展架构

> 一句话：Skill 做业务，Plugin 做功能，Extension 改流程。

---

## 一、三者总览

| 层级 | 定位 | 例子 |
|---|---|---|
| **Skill** | 面向用户场景的业务能力 | 查订单、写报告、办审批 |
| **Plugin** | 面向功能执行的原子工具 | 查数据库、调接口、读文件 |
| **Extension** | 面向框架生命周期的扩展钩子 | 请求过滤、日志、脱敏 |

```
用户请求
   │
   ▼
Extension（前置处理：过滤、风控、日志）
   │
   ▼
Skill（匹配业务场景）
   │
   ▼
Plugin（执行原子操作）
   │
   ▼
Extension（后置处理：脱敏、格式化、日志）
   │
   ▼
返回用户
```

---

## 二、Skill（技能）

### 1. 定义

**一句话**：Skill 是 Agent 面向终端用户的最小完整业务单元。

### 2. 组成

```
Skill = 元信息 + 意图规则 + 参数模板 + Plugin 列表 + 结果处理
```

| 组成 | 说明 |
|---|---|
| 元信息 | 名称、描述、版本、状态 |
| 意图规则 | 触发该 Skill 的关键词、语义规则 |
| 参数模板 | 需要提取的参数、缺参追问逻辑 |
| Plugin 列表 | 需要调用的底层插件 |
| 结果处理 | 数据格式化、话术封装、异常兜底 |

### 3. 示例

```
Skill：查询订单
- 意图规则："查订单"、"我的订单"、"订单状态"
- 参数：用户 ID、订单号
- 依赖 Plugin：get_user、get_order
- 结果：返回订单状态话术
```

---

## 三、Plugin（插件）

### 1. 定义

**一句话**：Plugin 是 Agent 的底层原子工具，只做单一功能，不封装完整业务。

### 2. 特点

- 无业务语义
- 可被多个 Skill 复用
- 热插拔

### 3. 示例

```
Plugin：get_user       # 查用户信息
Plugin：get_order      # 查订单信息
Plugin：send_email     # 发邮件
Plugin：execute_sql    # 执行 SQL
```

### 4. Skill vs Plugin

| 对比 | Skill | Plugin |
|---|---|---|
| 层级 | 业务层 | 工具层 |
| 粒度 | 粗，完整场景 | 细，单一功能 |
| 是否面向用户 | 是 | 否 |
| 复用性 | 低 | 高 |
| 关系 | 组合多个 Plugin | 被 Skill 调用 |

---

## 四、Extension（扩展）

### 1. 定义

**一句话**：Extension 是 Agent 框架的生命周期钩子，用来扩展和管控框架流程。

### 2. 常见场景

| 阶段 | 作用 |
|---|---|
| **前置扩展** | 输入清洗、敏感词过滤、权限校验 |
| **Prompt 扩展** | 动态拼接 Prompt、注入规则 |
| **调度扩展** | 修改工具调用优先级 |
| **后置扩展** | 输出脱敏、格式统一、话术修正 |
| **监控扩展** | 日志埋点、指标统计、BadCase 采集 |

### 3. Extension vs Plugin

| 对比 | Plugin | Extension |
|---|---|---|
| 层级 | 业务工具层 | 框架底层 |
| 触发方式 | 被 Skill 主动调用 | 跟随框架生命周期自动触发 |
| 作用 | 扩展"能做什么" | 扩展"怎么运行" |
| 生效范围 | 按需使用 | 全局生效 |

---

## 五、三层协作流程

```
用户请求进入
   │
   ▼
Extension 前置处理
   │
   ▼
Skill 发现与匹配
   │
   ▼
Skill 调度 Plugin 执行
   │
   ▼
Plugin 返回原子结果
   │
   ▼
Skill 聚合结果
   │
   ▼
Extension 后置处理
   │
   ▼
输出给用户
```

## 六、面试高频问题

### Q1：Skill、Plugin、Extension 的区别？

**答**：Skill 是面向用户的业务技能；Plugin 是底层原子工具，被 Skill 调用；Extension 是框架生命周期钩子，全局增强流程。

### Q2：新增一个查天气功能，应该加在哪一层？

**答**：加 Plugin 层实现天气接口调用；如果需要封装完整对话流程，再在外层加 Skill。

### Q3：全局日志和脱敏应该加在哪一层？

**答**：用 Extension，在请求前置和响应后置阶段统一处理。

### Q4：Skill 和 Plugin 的复用性为什么不同？

**答**：Skill 面向具体业务场景，耦合度高；Plugin 是原子功能，多个 Skill 可以共用。

### Q5：为什么要分层？

**答**：业务层和工具层解耦，框架层和业务层解耦。新增业务只加 Skill，新增通用能力只加 Plugin，全局治理用 Extension，互不干扰。

---

## 七、项目口述模板

> 你项目中的 Agent 是怎么做能力扩展的？

我项目采用 Skill/Plugin/Extension 三层扩展架构。Skill 是最上层业务封装，比如查订单、生成报告，每个 Skill 包含意图匹配、参数解析、Plugin 调度和结果话术；Plugin 是底层原子工具，比如查数据库、调接口、发邮件，供多个 Skill 复用；Extension 是框架层钩子，做请求过滤、安全校验、日志埋点、输出脱敏等全局治理。这样业务迭代和框架增强互不干扰，新增功能只需要新增 Skill 或 Plugin。

---

## 八、一句话总结

- **Skill：用户要什么**
- **Plugin：具体怎么做**
- **Extension：流程怎么管**
- **分层目的：解耦、复用、可治理**
