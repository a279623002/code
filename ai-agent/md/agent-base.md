## 一、Agent 核心原理与整体工作逻辑（开篇必问）

### 1、什么是 AI Agent？和普通大模型对话的本质区别？

**标准答案**：

AI Agent 是基于大模型作为**推理大脑**，具备自主感知、思考规划、工具调用、迭代执行、自我纠错的任务自主执行体。

它和普通LLM对话的核心区别：

普通大模型是**被动单轮生成**，用户问一句答一句，无法自主拆解任务、无法主动调用工具、无法多轮迭代纠错，只能依赖已有知识；

Agent 是**主动闭环执行**，可以自主分析任务难度、拆解子步骤、判断是否需要工具、循环执行、根据结果修正决策，最终自主完成复杂完整任务。

### 2、Agent 的核心工作原理与核心三要素？

**标准答案**：

Agent 的核心原理是**大模型推理 \+ 外部工具能力 \+ 上下文记忆**，三要素构成完整智能体。

第一，**思考能力（LLM大脑）**：负责意图理解、任务拆解、逻辑推理、决策判断、纠错反思；

第二，**行动能力（工具Action）**：通过Function Call、插件、外部API实现真实操作，弥补模型知识滞后、能力有限的问题；

第三，**记忆能力（Context/Memory）**：通过上下文窗口、短期记忆、长期向量记忆，维持多轮任务连续性，实现多步骤接力执行。

整体工作逻辑：**感知需求 → 思维推理 → 决策行动 → 接收反馈 → 迭代修正 → 任务闭环**。

### 3、Agent 为什么能做复杂任务？核心底层逻辑是什么？

**标准答案**：

普通LLM的短板是**只能直接输出答案，不会分步思考、不会拆解任务、不会验证纠错**。

Agent 通过引入**结构化推理策略**，把“一步生成答案”改成“多步思考、分步执行、动态决策”，让模型模拟人类解决复杂问题的思路：先分析、再规划、再行动、再复盘，从而具备复杂任务自主解决能力。

主流推理策略包含三大核心：**CoT思维链、ReAct推理链、Plan\-Execute规划执行架构**。

## 二、CoT 思维链（Chain of Thought）基础核心

### 4、什么是 CoT 思维链？核心作用是什么？

**标准答案**：

CoT思维链是让大模型**分步输出推理过程**的提示词策略，不直接给出最终答案，而是先拆解逻辑、分步推导、最后汇总结果。

核心价值：解决大模型**复杂逻辑推理弱、一步输出容易错、不会拆解问题**的缺陷，大幅提升数学推理、逻辑分析、多步骤任务的准确率。

### 5、CoT 的工作逻辑与优缺点？

**标准答案**：

工作逻辑：用户输入复杂问题 → 模型自主拆分推理步骤 → 逐步骤逻辑推导 → 整合所有步骤输出最终结论。

优点：极大增强模型逻辑推理能力、减少跳跃性错误、思路可解释、可追溯；

缺点：**只有思考、没有行动**，纯文本推理，无法调用外部工具、无法获取实时数据、无法解决实操类任务，只能处理纯逻辑问答。

总结：**CoT 解决的是“想明白”的问题，解决不了“做出来”的问题**。

## 三、ReAct 推理链（Agent 工业级核心策略）

### 6、什么是 ReAct？和 CoT 的本质区别？

**标准答案**：

ReAct 是目前工业级 Agent 最主流的**思考\+行动联动推理框架**，全称 Reason \+ Act，推理与行动交替执行。

CoT 只有纯文本思考，ReAct 在 CoT 思考的基础上，增加了**外部行动执行**，是真正能落地的智能体策略。

核心区别：

CoT：只思考、不行动，纯内部逻辑推演；

ReAct：先思考、再行动、再观察、再迭代，**思考和外部动作交替闭环**。

### 7、ReAct 完整工作链路？

**标准答案**：

ReAct 固定四步闭环，也是所有工具型 Agent 的标准执行逻辑：

**第一步 Thought（思考）**：模型分析当前任务，判断现有信息是否足够，是否需要调用工具、需要什么工具、需要什么参数；

**第二步 Action（行动）**：执行 Function Call，调用对应插件、接口、知识库，发起外部操作；

**第三步 Observation（观察）**：接收工具返回的真实结果、数据、状态；

**第四步 Iteration（迭代）**：模型基于最新结果重新思考，判断任务是否完成、是否需要继续调用工具、是否需要修正参数，直至任务结束。

一句话总结：**CoT 是脑内推演，ReAct 是知行合一**。

### 8、ReAct 解决了 Agent 什么核心痛点？

**标准答案**：

解决了 CoT 纯推理无法落地的问题：

第一，解决模型知识陈旧、信息有限的问题，通过行动获取实时真实数据；

第二，解决复杂任务无法分步落地的问题，通过多轮迭代逐步推进；

第三，解决幻觉问题，所有结论基于工具真实返回，而非模型脑补；

第四，解决单一能力局限，让模型可以联动多工具、多接口完成复合任务。

## 四、Plan\-Execute 规划执行架构（复杂长任务高阶策略）

### 9、什么是 Plan\-Execute（规划\-执行）架构？适用场景？

**标准答案**：

Plan\-Execute 是面向**长周期、多步骤、高复杂、强依赖**任务的高阶 Agent 架构，分为顶层规划层和底层执行层，彻底解决 ReAct 长任务逻辑混乱、步骤跑偏、循环冗余的问题。

ReAct 是「边想边做」，适合短任务；

Plan\-Execute 是「先整体规划、再分步执行」，适合复杂长任务。

### 10、Plan\-Execute 完整工作逻辑？

**标准答案**：

分为两层解耦架构，完全标准化闭环：

**第一层：Plan 规划层（顶层大脑）**

接收用户复杂任务后，先整体拆解全局子任务清单、定义执行顺序、梳理依赖关系、明确终止条件、规避冲突步骤，生成完整任务执行蓝图，不做具体执行。

**第二层：Execute 执行层（动作单元）**

严格按照规划蓝图，逐个子任务执行 ReAct 迭代，完成工具调用、数据查询、结果汇总；每完成一步自动校验进度，同步状态给规划层。

**动态复盘修正**：执行中遇到异常、数据缺失、任务变更时，回传给规划层，动态更新任务蓝图，再继续执行。

### 11、Plan\-Execute 和 ReAct 的层级区别、选型场景？

**标准答案**：

**ReAct**：轻量即时推理，边思考边执行，无全局规划，适合短平快、步骤简单、依赖少的单次工具任务；缺点是长任务容易局部最优、全局混乱、无限循环、步骤遗漏。

**Plan\-Execute**：先全局统筹、再分步落地，有完整任务蓝图、有进度管控、有动态修正，适合多步骤嵌套、依赖复杂、周期长、精度要求高的复杂业务任务。

工程选型：**简单任务用 ReAct 轻量化迭代，复杂业务任务用 Plan\+ReAct 组合架构**。

## 五、三大策略横向对比（面试高频抢答）

### 12、CoT / ReAct / Plan\-Execute 一句话核心区分？

**标准答案**：

**CoT 思维链**：纯文本分步推理，只会想、不会做，解决逻辑推理问题；

**ReAct 推理链**：思考\+行动交替迭代，知行合一，解决中小型工具任务；

**Plan\-Execute 架构**：先全局规划、再分步执行，统筹\+落地结合，解决大型复杂长任务。
